## 一、双算法与 sheet 对应关系

**结论：跑哪个 sheet，config.yaml 就配对应算法，最后恢复的也是对应算法的主公钥。**

| sheet | config.yaml auth.algorithm | signer 签名密钥 | 最后恢复公钥 |
|-------|---------------------------|-----------------|-------------|
| pk_rsa | RSAWithSHA256 | RSA key1 | RSA key1 |
| pk_sm2 | SM2WithSM3 | SM2 key1 | SM2 key1 |

- 不要跨算法恢复（pk_sm2 结尾不能恢复 RSA key1）
- 后续 7.1/7.2 跑哪个算法，就接着对应 sheet 的结束状态继续

## 二、密钥管理的两套体系（不要混淆）

| 用途 | 位置 | 说明 |
|------|------|------|
| **运行时签名** | signer.py (硬编码) + cert/*.pem (优先) | 框架发请求时自动签名，**不要改动加载逻辑** |
| **testdata 引用** | data/keys.json + `${keys.xxx}` 语法 | Excel 测试数据中引用公钥值，由 excel_handler 替换 |

signer.py 只负责签名，keys.json 只负责提供测试数据中的公钥字符串。两者独立，key1 恰好相同只是因为签名用的就是 key1 的私钥。

## 三、指纹是确定性的

同一公钥永远产生相同指纹，必须写入断言精确匹配：

**RSA (SHA256 → BASE64):**
- key1: `/WiF31yZx0q9nF0fQBudD7xSeivFBfdEhG8hl/KQo1c=`
- key2: `8CPhhUBOk75PaceSD6UjfghIh2V0xEo+1v2nUN4k7uc=`
- key3: `EZe+Or1pc78hK6RoKjqYPGuZ7h59ohDZqJBa7xBCgb8=`

**SM2 (SM3 → BASE64):**
- key1: `mdg2R+NkXUI3WidsEba3Dkg/MBx98JaXO23L9XF8xK8=`
- key2: `RSiUBOLEkAC8AzPRie3UiN8UPHlrLyA0SVpYGEO7tUc=`
- key3: `gr1fZe+LtAM9IA5QhUWOj7Wm0fCavaBXn5oHLS/9UCM=`

计算方式：
- RSA: `BASE64( SHA256( 公钥BASE64字符串.encode("utf-8") ) )`
- SM2: `BASE64( SM3( bytes.fromhex(公钥HEX字符串) ) )`

## 四、configCHSMPk 业务规则

1. 最多 2 个公钥，**按内容去重**（不是按次数）
2. 首次配置 guest 权限即可，后续需要 trusted
3. 请求中 `algorithm` 是密钥算法 (`rsa` / `sm2`)
4. getCHSMPk 返回的 `result.algorithm` 是哈希算法 (`sha256` / `sm3`)
5. 混合算法可以配置（1个RSA + 1个SM2 = 2个，成功）
6. clearCHSMPk 清空**所有**公钥（二期明确，不是清单个）

## 五、已踩的坑

### 5.1 signer.py 密钥加载反复修改
**问题**: 多次在硬编码字符串、JSON读取、PEM文件之间来回改。
**结论**: PEM文件优先 → 硬编码兜底，两种方式并存。已验证通过，不要再动。

### 5.2 pk_sm2 超限测试用了已有的 key
**问题**: PK_SM2_007 原本用 key2（step 5 已配置），去重后实际是幂等200，不是超限400。
**修复**: keys.json 补了 SM2 key3，PK_SM2_007 和 PK_SM2_016 改用 key3。

### 5.3 pk_sm2 结尾被错误改成恢复 RSA key1
**问题**: 以为"后续模块"统一用RSA，把 pk_sm2 最后恢复改成了 RSA key1。
**结论**: 跑 sm2 sheet 时 config 就是 SM2 签名，最后恢复的当然是 SM2 key1。已回退。

## 六、getCHSMPk 响应格式

```json
{
    "status": 200,
    "message": "success",
    "timestamp": "...",
    "requestId": "...",
    "costMillis": 500,
    "result": {
        "algorithm": "sha256",       // 或 "sm3"
        "fingerprints": ["fp1", "fp2"]
    }
}
```

- configCHSMPk / clearCHSMPk 成功时无 result 字段
- 清空后 getCHSMPk: fingerprints 为空数组
- 清空后调 trusted 接口: 返回 403

## 七、7.3 testdata 覆盖矩阵

pk_rsa / pk_sm2 各 35 条（25步场景 + 10条异常）:

**场景步骤:**
1. 首次配置 1 个公钥 → 验证 1 个指纹
2. 幂等（重复配相同公钥）→ 仍 1 个
3. 增量超限（已有1 + 传2 = 3）→ 400，仍 1 个
4. 追加第 2 个 → 验证 2 个指纹
5. 逐个超限（已有2 + 传1）→ 400，仍 2 个
6. 跨算法超限（已满2 + 传另一算法）→ 400
7. 清空 → 验证为空 → trusted 接口 403
8. 混合算法（1个本算法 + 1个对方算法）→ 验证 2 个指纹
9. 一次传 3 个 → 400
10. 一次传 2 个 → 成功 → 去重验证
11. 恢复主公钥 → 验证就绪

**异常用例 (E01-E10):**
非法algorithm、空pks、格式错误公钥、缺必填字段、混合有效/无效公钥、各接口缺requestId

## 八、7.1/7.2 testdata 设计决策

### 8.1 不按算法拆行，跑两遍覆盖

测试数据只考虑参数校验和业务逻辑，不考虑签名算法。同一个 sheet (trusted) 切换 config.yaml 算法跑两遍（RSA / SM2）就覆盖了。因此 SM2 签名专用的测试用例（CHSM_INFO_006、CHSM_NET_005 等）不需要在 testdata 中生成单独的行。

### 8.2 Sheet 组织

| Sheet | 内容 | auth | 行数 |
|---|---|---|---|
| trusted | 所有7.1/7.2接口的参数校验+业务逻辑 | config.yaml enabled:true | 176行(166 enabled + 10 disabled) |
| guest | 3个无需认证的状态查询接口 | config.yaml enabled:false | 12行(9 enabled + 3 disabled) |

### 8.3 VSM 场景编排 (SCN_VSM)

VSM 接口全部需要 vsmId，通过场景编排解决：
- step 1 (setup): getCHSMInfo → `save_vars: vsmId=result.vsmIds[0]`
- step 2~19: VSM 各接口测试，json_data 中用 `${vsmId}` 引用

enabled=no 的步骤（VSM_IMPORT_001 step 10, VSM_RESET_001 step 17, VSM_UPGRADE_001 step 19）会被 ExcelHandler 过滤，不影响场景连续性，步骤号跳跃不影响排序。

### 8.4 enabled=no 汇总

| 类型 | 用例 | 原因 |
|---|---|---|
| 待确认sign值 | CHSM_IMPORT_001/003/004, CHSM_UPGRADE_001/003, CHSM_RESTORE_001, VSM_IMPORT_001, VSM_UPGRADE_001 | 缺真实数据包签名 |
| 高危操作 | CHSM_RESTART_001, VSM_RESET_001 | 重启/重置风险 |
| 缺vsmId | GUEST_VSM_STATUS_001 | guest 场景下无法动态获取 |

### 8.5 结果回填机制

- `result_collector.py` 收集全量结果（pass/fail/error），保存到 `output/test_results.json`
- `issue_collector.py` 只收集失败，保存到 `output/issues.json`
- 两者都支持合并去重：同 case_id/ref_case_id 覆盖旧记录，跨 sheet 运行结果累积
- `tools/backfill_results.py` 读取 test_results.json，按 ref_case_id 回填到测试用例 Excel

## 九、Section 8/9 testdata 设计决策

### 9.1 Sheet 组织

Section 8（FileServer）和 Section 9（租户密评）合并到 `section8_9` sheet（57行），因为：
- 两者都需要 host 列覆盖（FileServer 用 `${fileServerHost}`，VSM 用 `${vsmHost}`）
- 不影响已有 trusted/guest sheet
- 可以独立运行，切换 config 中的 sheet_name 即可

### 9.2 Section 8 FileServer 全部 enabled=no

FileServer 4 个接口全部标记 enabled=no（共 15 条），原因：
- 上传（8.1.1）：multipart 文件上传，框架不支持
- 获取描述/下载/删除（8.1.2-4）：依赖先有上传成功的文件
- 需要手动搭建 FileServer 环境后逐条启用

### 9.3 Section 9 SCN_TENANT 场景设计

用一个 40 步场景覆盖全部 8 个租户密评接口（33 enabled + 7 disabled）：

**业务关联链（steps 1-8）**：
1. getStatus 获取基线状态
2. authPK 配置 RSA 公钥 → 3. getAuthPKFingerprints 验证
4. authPK 追加 SM2 公钥 → 5. getAuthPKFingerprints 验证双算法
6. cleanPK 清除 → 7. getAuthPKFingerprints 验证清空
8. authPK 恢复公钥

**各接口异常用例（steps 9-37）**：
- authPK: 超限/alg无效/pks空/缺字段/格式错误/一次传3个
- getAuthPKFingerprints: 缺requestId/requestId空
- cleanPK: 缺requestId/requestId空
- getStatus: 无参数调用
- initKey: 正常(高危enabled=no)/缺requestId/requestId空
- exportBackupKeys: 正常(binary返回enabled=no)/缺字段/空字段
- importBackupKeys: 全部enabled=no(文件上传)
- doVsmInit: 正常(高危enabled=no)/缺字段/无效值

**场景清理（steps 38-40）**：cleanPK → 恢复主公钥 → 验证

### 9.4 Section 9 认证机制

- 配置公钥指纹（authPK）和清除公钥指纹（cleanPK）**不需要认证**（guest）
- 其他接口（initKey/exportBackupKeys/importBackupKeys/doVsmInit/getStatus）需要 CHSM 认证
- 认证 header 同 7.1/7.2：CHSM-AuthPK / CHSM-SignatureAlg / CHSM-Signature
- host 为 VSM IP:port（`${vsmHost}`），非管理接口的 base_url

### 9.5 Servlet URL 模式

Section 9 使用 servlet 风格 URL，通过 params 列传递 method 参数：
- `/authServlet?method=authPK` → params="method=authPK"
- `/platformServlet?method=getStatus` → params="method=getStatus"
- POST 请求同时带 query params 和 json body

### 9.6 Section 8+9 enabled=no 汇总

| 类型 | 用例 | 原因 |
|---|---|---|
| multipart上传 | FS_UPLOAD_001~006 | 框架不支持 multipart |
| 依赖文件 | FS_GETINFO_001~003, FS_DOWNLOAD_001~003, FS_DELETE_001~003 | 需先上传成功 |
| 高危操作 | TENANT_INITKEY_001, TENANT_VSMINIT_001/002 | 初始化主密钥/VSM |
| binary返回 | TENANT_EXPORT_001 | 返回文件流无法断言 |
| 文件上传 | TENANT_IMPORT_001~003 | 需 multipart |
