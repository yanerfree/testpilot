# TestPilot 测试用例生成指导规范

## 1. 用例输出格式

所有生成的测试用例统一为 Markdown 表格，字段定义如下：

| 字段 | 说明 | 示例 |
|------|------|------|
| 用例ID | TC-模块缩写-三位序号 | TC-REG-001 |
| 标题 | "验证XXX功能/场景"，简洁明确 | 验证正确信息注册成功 |
| 模块 | 功能模块名称 | 用户注册、用户登录、登录页面 |
| 测试类型 | 见下方分类 | 功能测试、参数校验 |
| 优先级 | P0/P1/P2/P3 | P0 |
| 前置条件 | 执行前必须满足的条件，无则填"无" | 已注册用户 email=test@example.com |
| 测试步骤 | 编号列出具体操作，每步一个动作 | 1. 发送 POST /api/auth/login ... |
| 预期结果 | 编号对应步骤，明确可验证的结果 | 1. 状态码 200 2. 返回 token |

## 2. 测试类型分类

### API 接口测试

| 测试类型 | 含义 | 典型场景 |
|---------|------|---------|
| 功能测试 | 正常流程验证 | 合法参数请求成功，返回正确数据 |
| 参数校验 | 输入参数合法性验证 | 必填字段缺失、类型错误、格式不正确、空字符串 |
| 业务规则 | 业务逻辑约束验证 | 邮箱已注册(409)、新旧密码相同、数据状态冲突 |
| 异常场景 | 错误/异常条件验证 | 未认证(401)、token过期、资源不存在(404) |
| 边界条件 | 极端值/边界值验证 | 空请求体、超长字符串、特殊字符、最大/最小值 |

### E2E 页面测试

| 测试类型 | 含义 | 典型场景 |
|---------|------|---------|
| 功能测试 | 正常用户操作流程 | 填写表单 → 提交 → 成功跳转 |
| 表单校验 | 前端输入校验 | 必填项为空、邮箱格式错、密码长度不足 |
| 交互反馈 | UI 状态变化验证 | 按钮禁用/启用、loading、成功/失败提示 |
| 页面导航 | 页面跳转验证 | 链接跳转、操作后重定向、浏览器后退 |
| 异常场景 | 异常操作验证 | 密码错误、账号锁定、网络异常 |
| 边界条件 | 极端操作验证 | 空数据状态、重复提交、超长输入 |

## 3. 优先级定义

| 优先级 | 含义 | 覆盖场景 | 占比建议 |
|--------|------|---------|---------|
| P0 | 冒烟/核心 | 每个接口/页面的核心正常流程，必须通过 | 10-15% |
| P1 | 高优先级 | 主流程分支、关键异常（认证失败、必填校验） | 30-40% |
| P2 | 中优先级 | 边界条件、非关键异常、格式校验 | 30-40% |
| P3 | 低优先级 | 极端场景、性能边界、罕见路径 | 10-15% |

## 4. 覆盖要求

### API 接口 — 每个接口至少覆盖：

1. **正常流程 × 1条**（P0）：合法参数的成功请求，验证状态码和返回数据结构
2. **必填字段缺失 × N条**（P1）：每个必填字段各一条
3. **业务规则 × N条**（P1）：文档中描述的业务约束（重复、冲突、权限）
4. **认证异常 × 2条**（P1）：需认证接口必须覆盖无token和token无效
5. **参数格式 × N条**（P2）：邮箱格式、密码规则等文档中提到的校验
6. **空值/边界 × N条**（P2）：空字符串、空请求体、超长值

### E2E 页面 — 每个页面至少覆盖：

1. **正常流程 × 1条**（P0）：完整的用户操作到成功结果
2. **必填校验 × N条**（P1）：每个必填输入框各一条
3. **格式校验 × N条**（P1/P2）：文档中提到的格式校验
4. **异常反馈 × N条**（P1）：错误密码、账号不存在等
5. **页面导航 × N条**（P1）：每个链接/跳转各一条
6. **边界场景 × N条**（P2/P3）：空数据、重复提交

## 5. 编写规范

### 用例ID
- 格式：`TC-{模块缩写}-{三位序号}`
- 模块缩写从接口路径或页面名称推导，如 `/api/auth/register` → `REG`，"登录页面" → `LOGIN`
- 同模块内序号连续递增

### 标题
- 格式："验证 + 场景描述"
- 正面描述："验证正确信息注册成功"
- 反面描述："验证缺少 email 字段返回 400"
- 不超过 30 个字

### 前置条件
- 写清具体数据："已注册用户 email=test@example.com, password=Abc123456"
- 不写模糊描述："已有测试账号"
- 多个条件用编号列出
- 无前置条件写"无"

### 测试步骤
- 每步一个动作，用编号列出
- API 测试要明确：请求方法、路径、请求体、Header
- E2E 测试要明确：操作对象、输入值
- 不写："填写正确的信息"
- 要写："在邮箱输入框输入 test@example.com"

### 预期结果
- 编号对应步骤
- 可验证、可度量
- API：明确状态码、返回字段名和类型
- E2E：明确页面跳转路径、提示文案、元素状态
- 不写："跳转成功"
- 要写："页面跳转到 /dashboard"

## 6. 用例独立性原则

- 每条用例可独立执行，不依赖其他用例的执行顺序
- 前置条件中写清所有依赖数据，不假设其他用例已执行
- 有数据依赖的用例，在前置条件中说明如何准备数据
